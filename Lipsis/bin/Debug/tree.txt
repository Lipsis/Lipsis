   <h2 id="abstract">
    <span>
     Abstract
    </span>
   </h2>
   <div property="dc:abstract" datatype="">
    <p>
     <span>
      The 
     </span>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       Basic document type includes the minimal set of modules required to be an 
     </span>
     <acronym title="Extensible HyperText Markup 
Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       host language document type, and in addition it includes images, forms, basic tables, and object support. It is designed for Web clients that do not support the full set of
     </span>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       features; for example, Web clients such as mobile phones, 
     </span>
     <acronym title="Personal Digital Assistant">
      <span>
       PDA
      </span>
     </acronym>
     <span>
      s, pagers, andset top boxes. The document type is rich enough for content authoring.
     </span>
    </p>
    <p>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       Basic is designed as a common base that may be extended. The goal of 
     </span>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       Basic is to serve as a common language supported by various kinds of user agents.
     </span>
    </p>
    <p>
     <span>
      This revision, 1.1 Second Edition, supercedes version 1.1 as defined in 
     </span>
     <a href="http://www.w3.org/TR/2008/REC-xhtml-basic-20080729">
      http://www.w3.org/TR/2008/REC-xhtml-basic-20080729
     </a>
     <span>
      . In thisrevision, an XML Schema implementation and the 
     </span>
     <code>
      <span>
       lang
      </span>
     </code>
     <span>
       attribute have been added. In the update from version 1.0 to version 1.1, several new features were incorporated into the language inorder to better serve the small-device community that is this language's major user:
     </span>
    </p>
    <ol>
     <li>
      <span>
       XHTML Forms (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       Intrinsic Events (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       The value attribute for the 
      </span>
      <code>
      </code>
      <span>
        element (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       The target attribute (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       The style element (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       The style attribute (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       XHTML Presentation module (defined in [
      </span>
      <a href="#ref_xhtmlmod">
       XHTMLMOD
      </a>
     </li>
     <li>
      <span>
       The inputmode attribute (defined in 
      </span>
      <a href="#s_inputmode">
       Section 5
      </a>
      <span>
        of this document)
      </span>
     </li>
    </ol>
    <p>
     <span>
      The document type definition is implemented using 
     </span>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       modules as defined in "
     </span>
     <cite>
      <acronym title="Extensible HyperText Markup 
Language">
       <span>
        XHTML
       </span>
      </acronym>
      <span>
        Modularization
      </span>
     </cite>
     <span>
      " [
     </span>
     <a href="#ref_xhtmlmod">
      XHTMLMOD
     </a>
    </p>

   <h2 typeof="bibo:Chapter" about="#s_intro" id="s_intro">
    <span>
     1. Introduction
    </span>
   </h2>
   <h3 typeof="bibo:Chapter" about="#s1.1" id="s1.1">
    <span>
     1.1. 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      for Small Information Appliances
    </span>
   </h3>
   <p>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4 is a powerful language for authoring Web content, but its design does not take into consideration issues pertinent to small devices,including the implementation cost (in power, memory, 
    </span>
    <abbr title="et cetera" xml:lang="la">
     <span>
      etc.
     </span>
    </abbr>
    <span>
     ) of the full feature set. Consumer devices with limited resources cannot generally afford toimplement the full feature set of 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4. Requiring a full-fledged computer for access to the World Wide Web excludes a large portion of thepopulation from consumer device access of online information and services.
    </span>
   </p>
   <p>
    <span>
     Because there are many ways to subset 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
     , there are many almost identical subsets defined by organizations and companies. Without a commonbase set of features, developing applications for a wide range of Web clients is difficult.
    </span>
   </p>
   <p>
    <span>
     The motivation for 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic is to provide an 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      document typethat can be shared across communities (
    </span>
    <acronym title="exempli gratia" xml:lang="lt">
     <span>
      e.g.
     </span>
    </acronym>
    <span>
      desktop, 
    </span>
    <acronym title="Television">
    </acronym>
    <span>
     , and mobile phones), and that is rich enough tobe used for simple content authoring. New community-wide document types can be defined by extending 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic in such a way that
    </span>

   <h3 typeof="bibo:Chapter" about="#s1.2" id="s1.2">
    <span>
     1.2. Background and Requirements
    </span>
   </h3>
   <p>
    <span>
     Information appliances are targeted for particular uses. They support the features they need for the functions they are designed to fulfill. The following are examples of different informationappliances:
    </span>
   </p>
   <ul>
    <li>
     <span>
      Mobile phones
     </span>
    </li>
    <li>
     <span>
      Televisions
     </span>
    </li>
    <li>
     <acronym title="Personal Digital Assistant">
      <span>
       PDA
      </span>
     </acronym>
    </li>
    <li>
     <span>
      Vending machines
     </span>
    </li>
    <li>
     <span>
      Pagers
     </span>
    </li>
    <li>
     <span>
      Car navigation systems
     </span>
    </li>
    <li>
     <span>
      Mobile game machines
     </span>
    </li>
    <li>
     <span>
      Digital book readers
     </span>
    </li>
    <li>
     <span>
      Smart watches
     </span>
    </li>
   </ul>
   <p>
    <span>
     Existing subsets and variants of 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      for these clients include Compact 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <a href="#ref_chtml">
     CHTML
    </a>
    <span>
     ], the Wireless Markup Language [
    </span>
    <a href="#ref_wml">
     WML
    </a>
    <span>
     ], and the "
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4.0 Guidelines for Mobile Access" [
    </span>
    <a href="#ref_mhtml">
     GUIDELINES
    </a>
    <span>
     ]. The common features found in these document types include:
    </span>
   </p>
   <ul>
    <li>
     <span>
      Basic text (including headings, paragraphs, and lists)
     </span>
    </li>
    <li>
     <span>
      Hyperlinks and links to related documents
     </span>
    </li>
    <li>
     <span>
      Basic forms
     </span>
    </li>
    <li>
     <span>
      Basic tables
     </span>
    </li>
    <li>
     <span>
      Images
     </span>
    </li>
    <li>
     <span>
      Meta information
     </span>
    </li>
   </ul>
   <p>
    <span>
     This set of 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      features has been the starting point for the design of 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
     Basic. Since many content developers are familiar with these 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      features, they comprise a useful host language that may be combined with markupmodules from other languages according to the methods described in "
    </span>
    <cite>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       Modularization
     </span>
    </cite>
    <span>
     " [
    </span>
    <a href="#ref_xhtmlmod">
     XHTMLMOD
    </a>
    <span>
     ]. For example, 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic may be extended with a custom module to support richer markup semantics inspecific environments.
    </span>
   </p>
   <p>
    <span>
     It is not the intention of 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic to limit the functionality of future languages. But since the features in 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4 (frames, advanced tables, 
    </span>
    <abbr title="et cetera">
     <span>
      etc.
     </span>
    </abbr>
    <span>
     ) were developed for a desktop computer type of client, they have proved to be inappropriatefor many non-desktop devices. 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic will be extended and built upon. Extending 
    </span>
    <acronym title="Extensible HyperText Markup 
Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      from a common and basic set of features, instead of almost identical subsets or the too-large set of functions in 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
     4, will be good for interoperability on the Web, as well as for scalability.
    </span>
   </p>
   <p>
    <span>
     Compared to the rich functionality of 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4, 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic may look like onestep back, but in fact, it is two steps forward for clients that do not need what is in 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      4 and for content developers who get one 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      subset instead of many.
    </span>
   </p>

     1.3. Design Rationale
    </span>
   </h3>
   <p>

     This section explains why certain 
    </span>
    <acronym title="HyperText Markup Language">
     <span>
      HTML
     </span>
    </acronym>
    <span>
      features are not part of 
    </span>
    <acronym title="Extensible HyperText Markup Language">

     </span>
    </acronym>
    <span>
     Basic.

   <h4 typeof="bibo:Chapter" about="#s1.3.1" id="s1.3.1">
    <span>
     1.3.1. Presentation
    </span>
   </h4>
   <p>
    <span>
     Many simple Web clients cannot display fonts other than monospace. Bi-directional text, bold faced font, and other text extension elements are not supported.
    </span>
   </p>
   <p>
    <span>
     It is recommended that style sheets be used to create a presentation that is appropriate for the device.
    </span>
   </p>
   <h4 typeof="bibo:Chapter" about="#s1.3.2" id="s1.3.2">
    <span>
     1.3.2. Tables
    </span>
   </h4>
   <p>
    <a href="http://www.w3.org/TR/xhtml-modularization/abstract_modules.html#s_simpletablemodule">
      tables
    </a>
    <a href="#ref_xhtmlmod">
     XHTMLMOD
    </a>
    <span>
     ], section 5.6.1) are supported, but tables can be difficult to display on small devices. It is recommended that content developers follow the Web Content AccessibilityGuidelines 1.0 for 
    </span>
    <a href="http://www.w3.org/TR/1999/WAI-WEBCONTENT-19990505/#gl-table-markup">
     creating accessible tables
    </a>
    <a href="#ref_wai-webcontent">
     WCAG10
    </a>
    <span>
     ], Guideline 5). Note that inthe Basic Tables Module, nesting of tables is prohibited.
    </span>
   </p>
   <h4 typeof="bibo:Chapter" about="#s1.3.3" id="s1.3.3">
    <span>
     1.3.3. Frames
    </span>
   </h4>
   <p>
    <span>
     Frames are not supported. Frames depend on a screen interface and may not be applicable to some small appliances like phones, pagers, and watches.
    </span>

   <h3 typeof="bibo:Chapter" about="#s2.1" id="s2.1">
    <span>
     2.1. Document Conformance
    </span>
   </h3>
   <p>
    <span>
     A Conforming 
    </span>
    <acronym title="Extensible HyperText Markup Language">
     <span>
      XHTML
     </span>
    </acronym>
    <span>
      Basic document is a document that requires only the facilities described as mandatory in this specification. Sucha document must meet all of the following criteria:
    </span>
   </p>
   <ol>
    <li>
     <span>
      The document must conform to the constraints expressed in 
     </span>
     <a href="#a_dtd">
      Appendix B
     </a>
     <span>
       and 
     </span>
     <a href="#a_schema">
      Appendix C
     </a>
    </li>
    <li>
     <span>
      The root element of the document must be 
     </span>
     <code>
      <span>
       &lt;html&gt;
      </span>
     </code>
    </li>
    <li>
     <span>
      The name of the default namespace on the root element must be the 
     </span>
     <acronym title="Extensible HyperText Markup Language">
      <span>
       XHTML
      </span>
     </acronym>
     <span>
       namespace name, 
     </span>
     <code>
      <span>
       http://www.w3.org/1999/xhtml
      </span>
     </code>
     <span>
      . 
     </span>
     <p>
      <span>
       The start tag MAY also contain the declaration of the XML Schema Instance Namespace and an XML Schema Instance 
      </span>
      <code>
       <span>
        schemaLocation
       </span>
      </code>
      <span>
        attribute [
      </span>
      <a class="nref" href="#ref_XMLSCHEMA">
       XMLSCHEMA
      </a>
      <span>
       ]. Such an attribute would associate the XHTML namespace 
      </span>
      <code>
       <span>
        http://www.w3.org/1999/xhtml
       </span>
      </code>
      <span>
        with the XML Schema at the URI 
      </span>
      <code>
       <span>
        http://www.w3.org/MarkUp/SCHEMA/xhtml-basic11.xsd
       </span>
      </code>
     </p>
    </li>
    <li>
     <span>
      There must be a DOCTYPE declaration in the document prior to the root element. If present, the public identifier included in the DOCTYPE declaration must reference the 
     </span>
     <acronym title="Document 
Type Definition">
      <span>
       DTD
      </span>
     </acronym>
     <span>
       found in 
     </span>
     <a href="#a_dtd">
      Appendix B
     </a>
     <span>
       using its Formal Public Identifier. The system identifier may be modified appropriately. 
     </span>
     <pre class="dtd">
      &lt;!DOCTYPE html PUBLIC "-//W3C//DTD XHTML Basic 1.1//EN"    "http://www.w3.org/TR/xhtml-basic/xhtml-basic11.dtd"&gt;
     </pre>
    </li>
    <li>
     <span>
      The 
     </span>
     <acronym title="Document Type Definition">
      <span>
       DTD
      </span>
     </acronym>
     <span>
       subset must not be used to override any parameter entities in the 
     </span>
     <acronym title="Document Type Definition">
      <span>
       DTD
      </span>
     </acronym>
    </li>
   </ol>
   <p>
    <span>
     XHTML Basic 1.1 documents SHOULD be labeled with the Internet Media Type "application/xhtml+xml" as defined in [
    </span>
    <a class="nref" href="#ref_RFC3236">
     RFC3236
    </a>
    <span>
     ]. For further information on usingmedia types with XHTML, see the informative note [
    </span>
    <a class="nref" href="#ref_XHTMLMIME">
     XHTMLMIME
    </a>

   <p>
    <span>
     The attribute 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      provides a 
    </span>
    <em>
     <span>
      hint
     </span>
    </em>
    <span>
      to the user agent to select an appropriate input mode for the text input expected in an associated form control. The input mode may bea keyboard configuration, an input method editor (also called front end processor) or any other setting affecting input on the device(s) used.
    </span>
   </p>
   <p>
    <span>
     Using 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
     , the author can give hints to the agent that make form input easier for the user. Authors should provide 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attributes wherever possible, makingsure that the values used cover a wide range of devices.
    </span>
   </p>
   <h3 typeof="bibo:Chapter" about="#s_inputmodesyntax" id="s_inputmodesyntax">
    <span>
     5.1 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      Attribute Value Syntax
    </span>
   </h3>
   <p>
    <span>
     The value of the 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute is a white space separated list of tokens. Tokens are either sequences of alphabetic letters or absolute URIs. The later can be distinguished fromthe former by noting that absolute URIs contain a ':'. Tokens are case-sensitive. All the tokens consisting of alphabetic letters only are defined in this specification, in 
    </span>
    <a href="#s_mode-values">
     <b>5.3 List of Tokens
    </a>
    <span>
      (or a successor of this specification).
    </span>
   </p>
   <p>
    <span>
     This specification does not define any URIs for use as tokens, but allows others to define such URIs for extensibility. This may become necessary for devices with input modes that cannot becovered by the tokens provided here. The URI should dereference to a human-readable description of the input mode associated with the use of the URI as a token. This description should describe theinput mode indicated by this token, and whether and how this token modifies other tokens or is modified by other tokens.
    </span>
   </p>
   <h3 typeof="bibo:Chapter" about="#s_mode-ua-behavior" id="s_mode-ua-behavior">
    <span>
     5.2 User Agent Behavior
    </span>
   </h3>
   <p>
    <span>
     Upon entering an empty form control with an 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute, the user agent should select the input mode indicated by the 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute value. User agentsshould not use the 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute to set the input mode when entering a form control with text already present. To set the appropriate input mode when entering a form control thatalready contains text, user agents should rely on platform-specific conventions.
    </span>
   </p>
   <p>
    <span>
     User agents should make available all the input modes which are supported by the (operating) system/device(s) they run on/have access to, and which are installed for regular use by the user. Thisis typically only a small subset of the input modes that can be described with the tokens defined here.
    </span>
   </p>
   <div class="note">
    <p>
     <b>
      <span>
       Note:
      </span>
     </b>
    </p>
    <p>
     <span>
      Additional guidelines for user agent implementation are found at 
     </span>
     <a href="#ref_uaag-1.0">
      [UAAG 1.0]
     </a>
    </p>
   </div>
   <p>
    <span>
     The following simple algorithm is used to define how user agents match the values of an 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute to the input modes they can provide. This algorithm does not have to beimplemented directly; user agents just have to behave as if they used it. The algorithm is not designed to produce "obvious" or "desirable" results for every possible combination of tokens, but toproduce correct behavior for frequent token combinations and predictable behavior in all cases.
    </span>
   </p>
   <p>
    <span>
     First, each of the input modes available is represented by one or more lists of tokens. An input mode may correspond to more than one list of tokens; as an example, on a system set up for a Greekuser, both "greek upperCase" and "user upperCase" would correspond to the same input mode. No two lists will be the same.
    </span>
   </p>
   <p>
    <span>
     Second, the 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute is scanned from front to back. For each token 
    </span>
    <var>
    </var>
    <span>
      in the 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      attribute, if in the remaining lists of tokens representingavailable input modes there is any list of tokens that contains 
    </span>
    <var>
    </var>
    <span>
     , then all lists of tokens representing available input modes that do not contain 
    </span>
    <var>
    </var>
    <span>
      are removed. If there is noremaining list of tokens that contains 
    </span>
    <var>
    </var>
    <span>
     , then 
    </span>
    <var>
    </var>
    <span>
      is ignored.
    </span>
   </p>
   <p>
    <span>
     Third, if one or more lists of tokens are left, and they all correspond to the same input mode, then this input mode is chosen. If no list is left (meaning that there was none at the start) or ifthe remaining lists correspond to more than one input mode, then no input mode is chosen.
    </span>
   </p>
   <div class="example">
    <p>
     <span>
      Example: Assume the list of lists of tokens representing the available input modes is: {"cyrillic upperCase", "cyrillic lowerCase", "cyrillic", "latin", "user upperCase", "user lowerCase"}, thenthe following 
     </span>
     <code>
      <span>
       inputmode
      </span>
     </code>
     <span>
       values select the following input modes: "cyrillic title" selects "cyrillic", "cyrillic lowerCase" selects "cyrillic lowerCase", "lowerCase cyrillic" selects"cyrillic lowerCase", "latin upperCase" selects "latin", but "upperCase latin" does select "cyrillic upperCase" or "user upperCase" if they correspond to the same input mode, and does not select anyinput mode if "cyrillic upperCase" and "user upperCase" do not correspond to the same input mode.
     </span>
    </p>
   </div>

    <span>
     5.3.2 Modifier Tokens
    </span>
   </h4>
   <p>
    <span>
     Modifier tokens can be added to the scripts they apply in order to more closely specify the kind of characters expected in the form control. Traditional PC keyboards do not need most modifiertokens (indeed, users on such devices would be quite confused if the software decided to change case on its own; CAPS lock for upperCase may be an exception).    <h3 typeof="bibo:Chapter" about="#s_mode-patterns" id="s_mode-patterns">
    <span>
     5.4 Relationship to XML Schema pattern facets
    </span>
   </h3>
   <p>
    <span>
     User agents may use information available in an XML Schema pattern facet to set the input mode. Note that a pattern facet is a hard restriction on the lexical value of an instance data node, andcan specify different restrictions for different parts of the data item. Attribute 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      is a soft hint about the kinds of characters that the user may most probably start to inputinto the form control. Attribute 
    </span>
    <code>
     <span>
      inputmode
     </span>
    </code>
    <span>
      is provided in addition to pattern facets for the following reasons:
    </span>
   </p>
   <ol>
    <li>
     <p>
      <span>
       The set of allowable characters specified in a pattern may be so wide that it is not possible to deduce a reasonable input mode setting. Nevertheless, there frequently is a kind of characters thatwill be input by the user with high probability. In such a case, 
      </span>
      <code>
       <span>
        inputmode
       </span>
      </code>
      <span>
        allows to set the input mode for the user's convenience.
      </span>
     </p>
    </li>
    <li>
     <p>
      <span>
       In some cases, it would be possible to derive the input mode setting from the pattern because the set of characters allowed in the pattern closely corresponds to a set of characters covered by an
      </span>
      <code>
       <span>
        inputmode
       </span>
      </code>
      <span>
        attribute value. However, such a derivation would require a lot of data and calculations on the user agent.
      </span>
     </p>
    </li>
    <li>
     <p>
      <span>
       Small devices may leave the checking of patterns to the server, but will easily be able to switch to those input modes that they support. Being able to make data entry for the user easier is ofparticular importance on small devices.
      </span>
     </p>
    </li>
   </ol>
